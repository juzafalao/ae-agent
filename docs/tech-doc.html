Visão Geral do Sistema

O que é a LIA?

LIA é uma consultora de expansão de franquias movida por Inteligência Artificial. Ela atende leads no WhatsApp 24h por dia, conduz o funil de qualificação — do acolhimento ao diagnóstico, apresentação do modelo de negócios, envio de materiais e tratamento de objeções — até fazer a passagem para o especialista humano no momento certo do fechamento.

Objetivo principal

  • Qualificar leads de franquia 24/7 de forma automatizada
  • Manter conversa natural e humanizada via WhatsApp
  • Enviar materiais (PDFs) no momento certo do funil
  • Notificar especialista humano quando lead está pronto

Limites intencionais

  • LIA não fecha vendas — papel do especialista humano
  • Não negocia descontos ou condições de pagamento
  • Não agenda reuniões por conta própria
  • Não discute cláusulas de contrato em detalhe

Fluxo de Processamento Detalhado

1

Recebimento do webhook (Z-API → Express)

A Z-API faz um POST /webhook em tempo real ao receber mensagem. O servidor responde 200 OK imediatamente para evitar reenvios, depois processa de forma assíncrona via setImmediate().

2

Filtragem e validação (server.js)

Descarta: mensagens do próprio bot (fromMe), grupos (isGroup), números bloqueados (BLOCKED_PHONES), tipos não-texto (imagem/áudio/vídeo → envia aviso padrão), e eventos que não sejam ReceivedCallback.

3

Gerenciamento de histórico (memory.js)

Busca ou cria o histórico do número (Map em memória). Adiciona a mensagem do usuário. Mantém no máximo 20 mensagens — remove as mais antigas ao ultrapassar o limite.

4

Chamada à Claude API (agent.js)

Envia sistema: prompt.js (persona + dados da franquia + fluxo de venda). Envia histórico: últimas 20 mensagens formatadas como user/assistant. Modelo: claude-sonnet-4-20250514. Max tokens: 1024.

5

Parse da resposta e extração de tags

Claude retorna texto com possíveis tags de ação embutidas: [ENVIAR_APRESENTACAO] e [TRANSFERIR_LEAD]. O agent remove as tags do texto final e as executa separadamente.

6

Envio da resposta (zapi.js)

Texto é dividido em blocos de no máximo 1000 caracteres antes do envio. Múltiplos blocos são enviados em sequência com delay de 500ms para simular digitação humana.

7

Execução das ações

[ENVIAR_APRESENTACAO]: Envia 3 PDFs via sendDocument() (Apresentação, Plano de Negócios, COF oficial). [TRANSFERIR_LEAD]: Envia WhatsApp ao especialista com resumo da conversa e dados coletados do lead.

Componentes do Código

Arquivo Responsabilidade Destaques técnicos
src/server.js Ponto de entrada Express. Webhook handler. Health check. Responde 200 antes de processar; filtro BLOCKED_PHONES; setImmediate para não bloquear.
src/agent.js Orquestra chamada ao Claude, parse de tags, envio de resposta e ações. Regex para extração de tags; split de mensagens longas; delay entre múltiplos envios.
src/prompt.js Persona completa da LIA, dados da franquia, fluxo de venda em 7 etapas. Prompt do sistema carregado via env vars; não contém dados hardcoded sensíveis.
src/memory.js Histórico de conversas em memória RAM (Map). Chave = número de telefone; cap de 20 msgs; geração de resumo do lead para handoff.
src/zapi.js Camada de integração com Z-API (send text, document, image, audio, video, status). Client-Token opcional; conversão base64 para áudio; tratamento de erros com log.
.env Variáveis de ambiente sensíveis. Nunca commitado ao Git. Todas as credenciais e configurações mutáveis do ambiente.
railway.json Configuração de deploy no Railway. Força builder NIXPACKS; política de restart em falha; max 3 tentativas.

Modelo de Inteligência Artificial

Claude Sonnet 4 (Anthropic)

  • ID do modelo: claude-sonnet-4-20250514
  • Provider: Anthropic (anthropic.com)
  • SDK: @anthropic-ai/sdk
  • Max tokens por resposta: 1.024
  • Tipo de chamada: Messages API

Configuração do contexto

  • System prompt: persona + negócio completo
  • Histórico: últimas 20 msgs por conversa
  • Contexto mantido em memória RAM
  • Reiniciado apenas ao reiniciar o servidor
  • Lead summary: gerado automaticamente no handoff

Por que Claude Sonnet 4?

O Claude Sonnet 4 oferece o melhor equilíbrio entre capacidade conversacional e latência para aplicações de tempo real. Sua capacidade de seguir instruções complexas de persona e de manter coerência no funil de vendas ao longo de múltiplas mensagens é superior a modelos mais leves, enquanto sua velocidade de resposta permite uma experiência de chat fluida no WhatsApp.

🔒 Segurança de dados: Nenhuma conversa é armazenada em disco ou banco de dados. O histórico existe apenas em RAM enquanto o servidor está rodando. Ao reiniciar, o histórico é zerado. Não há PII em logs de produção.

Integrações Externas

Z-API (Gateway WhatsApp)

  • Tipo: SaaS — instância cloud dedicada
  • Autenticação: Instance ID + Token + Client-Token
  • Webhook configurado para: /webhook
  • Mensagens suportadas: texto, documento, imagem, áudio, vídeo
  • Eventos recebidos: ReceivedCallback (mensagens recebidas)
  • Painel: app.z-api.io

Anthropic API

  • Tipo: REST API com SDK Node.js oficial
  • Autenticação: API Key via header
  • Endpoint: Messages API (messages.create)
  • Cobrança: por token de entrada + saída
  • Latência típica: 2–4 segundos por resposta
  • Painel: console.anthropic.com

GitHub (controle de versão + CI/CD)

Repositório: juzafalao/ae-agent (privado recomendado). Cada push na branch main dispara automaticamente um novo deploy no Railway via integração nativa GitHub → Railway. Não há pipeline de testes configurado — deploys vão direto para produção.

Branch: main Auto-deploy: sim Zero downtime: sim (Railway rolling)

Infraestrutura de Hospedagem

Railway (PaaS — Platform as a Service)

O servidor LIA roda em um container gerenciado pelo Railway. Não há servidor físico ou VPS para administrar — Railway cuida de provisionamento, escalabilidade, certificado SSL e atualizações de runtime automaticamente.

Builder: Nixpacks (Node.js) Porta: dinâmica (injetada por Railway) SSL: automático (HTTPS) Restart: automático em falha Logs: em tempo real via dashboard

URLs de produção

  • Health check:
    https://ae-agent-production-355d.up.railway.app/health
  • Webhook Z-API:
    https://ae-agent-production-355d.up.railway.app/webhook

Variáveis de ambiente

  • Gerenciadas via painel Railway → Variables
  • Nunca hardcoded no código
  • Alterações exigem redeploy para surtir efeito
  • BLOCKED_PHONES: lista de números já atendidos
⚠️
Atenção — memória em RAM: O histórico de conversas é armazenado apenas em memória RAM. Se o servidor reiniciar (deploy, falha, etc.), todos os históricos são perdidos e as conversas começam do zero. Para persistência permanente, seria necessário adicionar um banco de dados (ex: Redis) — isso pode ser implementado em uma versão futura.

Escalabilidade e Multi-Marca

Padrão multi-marca

O sistema foi projetado para replicação: cada nova marca (franquia diferente) é uma cópia independente do projeto com seu próprio .env, src/prompt.js e instância Z-API. Cada instância roda em seu próprio serviço Railway, com deploy autônomo e cobrança separada.

Sem código compartilhado entre marcas Histórico e credenciais 100% isolados Deploy independente por marca

Capacidade atual

  • Conversas simultâneas: ilimitadas (assíncrono)
  • Histórico por lead: 20 mensagens em RAM
  • Latência de resposta: ~2–5 segundos
  • Limite Z-API: depende do plano contratado

Evoluções planejadas

  • Persistência de histórico (Redis ou banco SQL)
  • Dashboard de leads em tempo real
  • Pausa automática em horário de pico
  • Integração com CRM (Pipedrive, RD Station)

Segurança e Boas Práticas

O que está implementado

  • Credenciais apenas em variáveis de ambiente
  • .env no .gitignore — nunca vai para o Git
  • HTTPS obrigatório no Railway (SSL automático)
  • Filtro de mensagens do próprio bot (fromMe)
  • Filtro de grupos (isGroup)
  • BLOCKED_PHONES para proteção de leads ativos

Recomendações adicionais

  • Ativar Client-Token na Z-API (segurança extra no webhook)
  • Tornar repositório GitHub privado
  • Rotacionar API Keys periodicamente
  • Adicionar validação de origem do webhook (IP whitelist Z-API)
  • Monitoramento de gastos na Anthropic (budget alerts)
